Objective

This project will provide you with the opportunity to practice with interfaces as well as have some more practice with generic data structures, ArrayLists, and two-dimensional structures. The project is meant to be done in two phases; Phase 1 before the 3rd exam and Phase 2 after the 3rd exam.


Overview

In this project you will implement some elements needed for a theoretical Android game being developed. In Phase 1 you will build several classes that implement an interface called Battleable, and create a new one-dimensional generic data structure with certain rules for adding/removing things; that data structure will be able to hold any objects that extend the Battleable interface and will support a simple game of Solitaire. In Phase 2 you will add to the one-dimensional structure's abilities by supporting the exporting of data from it.

Choose Your Battle!

In this project scenario there is no GUI built yet because another team is working on that based on the design spec. When they complete their visual design they will make use of the classes you have built. The application architects have provided you with the JavaDocs for the classes, some initial acceptance tests (ExampleTests.java), and a wider array of acceptance tests (the ones on the submit server).


Phase 1

  • First, read through the Universe.java file to see what it provides and keep it in mind as you work on the project.
  • Next, read the documentation for the Battleable interface, which has been provided. In this project there will be at least three classes that implement this interface; PouchCreature, BattleCow, BattleFish.
  • Next, read through the BattleCow and BattleFish and PouchCreature classes and the method descriptions in the documentation and implement them as described.
  • Finally, implement the BattleDeck class (other than the two export methods export2Darray and export2Darraylist and the shuffle method) based on the method descriptions in the documentation. This class will support adding deep copies of Battleable objects to the structure. The structure is a double-ended list where new things are added to the two ends in alternating order. Additionally, the class supports two methods that the game designers have requested; the ability to have the Battleable objects at the two ends of the list fight each other in a game of Solitaire (returning the winner) and the ability to shuffle the deck.

Phase 2

  • Add the ability for your BattleDeck class to export the contents of the structure as a ragged two-dimensional structure; either using the Java array (export2Darray) or the ArrayList (export2Darraylist).
  • Add the ability to shuffle the deck the way the game designers requested.

 


Restrictions

  • While you are free to choose what you feel are the best internal representations for your data, the signatures of the public methods may not be altered.
  • Please remember that we will not be using any of the new features that Java 8 introduced and that in this course you are not allowed to try to use any new features that you might read about elsewhere when doing your assignments.

Project Submission
